Analysis, Reporting, and Feedback

  • Flare Project Analysis and Reports From the Analysis ribbon in Flare, you can scan your files and run reports to discover a wide variety of information about your project. This includes broken links or bookmarks, files with changes, topics not in a table of contents (TOC), and much more. You can also generate reports from the Project Organizer. See About Analysis and Reports.

  • Flare Text Analysis From the Tools ribbon in Flare, you can run text analysis on your files for things such as readability, average sentence length, and more. See Using Text Analysis.

  • MadCap Pulse Through the targets in your Flare projects, you can associate your output with MadCap Pulse. This lets you see valuable information about your output from the Pulse server dashboard. You can also add a comment section to the bottom of your online topics in order to communicate in real-time with end users who might have questions or feedback. See About MadCap Pulse.

  • Google Analytics You might decide to use Google Analytics (or another third-party tool) for reporting purposes on your output. If you have a Google Analytics account, you can obtain a tracking code, which you can then add to your master page in a Flare project. After publishing your output, you can use the Google Analytics website to see various reports about your output.

Tool Comparison

Any or all of these tools can provide you with valuable information. The information that you glean from Flare’s analysis feature is going to be somewhat different from the reports in the other tools. It’s best to research each of the methods to see which tool is returning the most valuable information for your needs. You might decide to use multiple tools.

What the MadCap Documentation Team Does

When we get close to a product release, we scan the files in our main “Shared” Flare project to discover the following information about our source files:

  • Critical Issues These are problems that must be fixed, such as broken links or bookmarks.
  • Files with Annotations We occasionally insert annotations in topics and snippets as reminders to do something before a topic is considered complete. Scanning the project for files with annotations lets us quickly locate these instances and take action if necessary.
  • Topics Without Concepts We use concept keywords in order to create search filter sets, as well as boost search results for certain topics. Therefore, we run this scan to make sure all necessary topics contain concept keywords.
  • Topics Not In Index We do not use an index for online output, but we do for some PDF targets. Also, index keywords can help boost search results for topics. So we run this scan to make sure we have inserted index keywords where necessary.
  • Unused Images After a release is finished, we run this scan to find obsolete images that can be removed from the project.

After a release is finished, we run the “Unused Images” scan on the relevant “Develop” child project (e.g., “Central-Develop,” “Lingo-Develop”) to find obsolete images that can be removed from the project. We like to run this scan on a child project instead of the main “Shared” project because it allows us to isolate the images that are intended only for that product. If we were to run the scan on our main project, we might see many unused images for other products that we may not really want to remove.

There are many other project analysis scans that you can run, but the ones listed above are those that we focus on.

Occasionally, we run text analysis on certain files, especially more complex ones, to make sure their readability level is acceptable.

And finally, our outputs are integrated with MadCap Pulse. We log in to our Pulse Dashboard to view certain reports related to the output to determine if we need to make changes (e.g., the “Search Phrases with No Results” report lets us know if we should create some synonyms in Flare).
