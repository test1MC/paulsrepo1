Topic Reviews

Most authors require some kind of a process for others to review their Flare documentation, as well as to contribute new content when necessary. There are several ways you can do this, depending on your resources and knowledge of your reviewers and contributors.

Following are the most common methods, although you might use another solution not listed here.

Flare to Central—Peer and SME Reviews

You can use the connection between Flare projects and MadCap Central for topic reviews. See About the Central Review Process.

Pros

Cons

It is an easy workflow.

Although snippets that are part of topics can be included in the review (or excluded from it), you will need to communicate with your reviewers about them. You can tell reviewers to open snippets separately in Central to provide edits and comments, or you can have reviewers insert comments above the read-only snippets in the topic itself. You might consider including larger and more substantial snippets as part of the review process, while excluding smaller snippets that you don’t feel need to be edited by reviewers.

You can send topics as well as snippets for review.

Flare to Flare—Peer Reviews

If you need a fellow Flare author to review content, a convenient solution is simply for both authors stay within the Flare project. There are a couple of ways to do this.

First, the original author can communicate (verbally, via email, etc.), asking another author to review the file(s). The reviewer opens the file(s), tracks his or her changes (see Enabling Track Changes), and makes edits and/or annotations. The reviewer lets the original author know the files have been reviewed. The original author opens those files and accepts or rejects the changes,.

A second way to use Flare for reviews between authors is to use the option in the Review ribbon to send files for review (in a review package). See Sending Files for Review.

After the reviewer finishes and returns the files, the original author can view the tracked changes, accepting or rejecting them as necessary. That author can then click a button to accept a reviewed file, replacing the original file with the reviewed one. See Accepting Reviewed Files.

Pros

Cons

By keeping everything within Flare, there is no need to transfer changes manually or convert file types. It is therefore more automated than some of other review solutions.

If you need multiple authors to review the same files, those files will need to be “daisy changed,” reviewed by only one person at a time.

Using a Flare-to-Flare workflow can work quite well if only one reviewer at a time is looking at a file.

In order to see tracked edits, you need to click an option on the Review tab to show changes.
Contributor to Flare—SME Reviews and Contributions

If you need a SME—rather than another Flare author—to review content, MadCap Contributor is one solution.

For reviews, you send files to the person in a review package. That individual opens Contributor, which is like a scaled-down version of Flare. Changes are automatically tracked, and the reviewer can also add annotations, then send the review package back to you. After accepting or rejecting changes in those files, you can accept them back into the Flare project, replacing the original source files. See Review Workflow—Sending to Others.

For new content, the person using Contributor can create the content and send it to you. You can then accept the file(s) into your Flare project. See Contribution Workflow.

Pros

Cons

Because Flare and Contributor read the same types of files, there is no need to transfer changes manually or convert files. It is more automated than some other review solutions.

A SME will need to have Contributor installed locally.

This solution can work quite well when only one individual at a time is looking at a file.

Although it works similarly to many other editors, there are some new tasks that a SME will need to learn. Also, because Contributor is used for both reviews and contributions, there may be many more features and options in the interface than a person will need to use.

 

The process can get complicated if you need to have multiple people review the same file. The files will need to be “daisy changed,” reviewed by one person at a time.
PDF—Peer and SME Reviews

Another option for both peer and SME reviews is to generate PDF output and send it to whomever you need to review the content. The reviewer then makes comments in the PDF and sends it back to you. After this, you manually make changes to the Flare files based on the PDF comments.

Pros

Cons

PDF is a format that is familiar to most people.

Reviewers cannot edit the PDF. They can only add comments.

 

There is no way to automatically transfer feedback into your source Flare files. It must all be done manually.

 

You will need to set up the PDF output in your Flare project, if it is not done already. And even if it is already done, the entire output might be much bigger than the small section that you need the person to review.
Word—Peer and SME Reviews and Contributions

This option for peer and SME reviews is similar to the one for PDF. With this solution, you generate Word output and send it to whomever you need to review the content. The reviewer then makes changes or comments in the Word document and sends it back to you. After this, you manually make changes to the Flare files based on the Word document

Pros

Cons

Word is a format that is familiar to most people.

There is no way to automatically transfer edits or comments into your source Flare files. It must all be done manually.

Reviewers can both make edits and add comments.

You will need to set up the Word output in your Flare project, if it is not done already. And even if it is already done, the entire output might be much bigger than the small section that you need the person to review.

Word targets in Flare have an option that lets you enable the review mode for the output. This way, reviewers do not need to remember to enable the Track Changes feature in the Word document. See Enabling Review Mode in Word Output.

 
What the MadCap Documentation Team Does

We use the review process available with MadCap Central, following these guidelines:

  • We limit the snippets that are included with reviews. It is more likely that we include larger snippets in a topic, while excluding many of the smaller snippets. We feel that including too many snippets as part of the review process can confuse reviewers, especially SMEs, who might not be familiar with snippets. So we communicate with reviewers to open and edit the snippets that we do include, but to insert annotations as necessary next to the read-only snippets that are excluded from the process.

  • Our reviews are sent from our “Develop” projects, rather than from our main “Shared” project. That’s because our “Develop” projects on Central include files that are in a state of development, whereas the “Shared” project on Central includes only published files. This is all a result of our branching process with source control. See Source Control.

    This means we currently only use about 90% of Central’s review process. The final part of the process—incorporating reviewed files back in to Flare—is done manually, rather than automatically. The reason for this is that we need to get the reviewed changes back into our “Shared” project, but the reviews are actually occurring in our child “Develop” projects. Once Central supports source control branching (allowing us to send reviews from our “Shared” project), we will be using 100% of the Central review process.

    However, although we use only 90% of the process, we decided that the benefits still outweigh the other review options, especially the ability to allow multiple individuals review the same file simultaneously.
