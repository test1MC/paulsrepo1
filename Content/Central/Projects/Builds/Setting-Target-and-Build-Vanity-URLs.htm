How to Set a Vanity for a Target

 How to Set a Vanity for a Build
  4. In the grid, click the check box to the left of the build.
  5. In the toolbar, click . The Edit Build Settings dialog opens.

  6. In the Vanity field, enter your custom vanity.

  7. (Optional) In this dialog, you can also set the build to Live or Keep. In addition, you can choose to exclude the output from search engines (see Excluding Live Builds from Search Engines).

  8. Click Save.

    If you have the Vanity column set to display in the builds grid, you will see your vanity listed in that build row. If you also have a target vanity set, the build vanity will display instead.

 Inheritance and Multiple URLs

Following are some important points to keep in mind when it comes to creating and using vanities:

  • Builds Inherit from Target If you set a vanity on the target, builds will inherit it. However, if you also set a vanity on a build, the vanity for the build will have precedence.
  • Use Build Vanity for Multiple Live Outputs You cannot have multiple builds set to “Live” using the same URL. Therefore, you can have multiple builds set to “Live” in a target only if you are using a unique vanity on a build.
  • One Live Target Vanity Per Target Only one of your builds can be set to “Live” using the target vanity. Suppose you have a target vanity and you set a build to “Live.” Then you set another build to “Live,” inheriting the target vanity. In that case, only the most recent build will be “Live” and the other will automatically be set to “Keep.”
 Example

Example

Let’s say your company has a product called FictionSoft Pro, and it consists of four modules. Your documentation team has therefore created four different Flare projects for each of those modules. Suppose that you are in charge of the Module 3 project.

Every year or so, your company releases a new version of the software. As a technical writer, you need to produce documentation for each of those releases.

In your Flare Module 3 project, you have an HTML5 target that you use to produce an online Help system. Instead of creating a different target for each release, you want to just keep using the same target.

You upload your Flare Module 3 project to Central, and you build output for FictionSoft Pro Version 1.

Your vanity for the license has already been set as “fictionsoft.” But you also want your Module 3 Version 1 documentation to have a unique path, different than the paths for other modules, and for later versions of Module 3. So you click the check box next to the build and click in the toolbar.

In the Edit Build Settings dialog, you click in the Vanity field and decide to enter mod3-online-v1. The first part of this vanity refers to the project name (mod3), to keep it distinct from the other projects. The second part refers to the kind of target (online), to keep it distinct from the many PDFs you will also be producing for that project. And the third part refers to the version number (v1), to keep it distinct from later versions.

You can also click Live to make the output available to your audience.

After clicking Save, you expand the build row and notice that the live URL has both the license vanity and the build vanity.

Sometime later, you are ready to publish output for Version 2. So you push your changes up to Central and build the target again. This time, you open the Edit Build Settings dialog and set the vanity to mod3-online-v2. You also set the build to Live.

When finished, you notice that the new build has its own unique URL, keeping it separate from the Version 1 output, which remains set to “Live” as well.

 Note

You can also change the vanity for your license. This includes the ability to edit the subdomain used in the URL, as well as mapping to your company’s host domain. See Changing the License Vanity URL.
